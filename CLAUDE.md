## Stack Context
- Language runtime: Python 3.14
- Orchestration: Dagster 1.x (software-defined assets, partitioned backfills)
- Transformation: dbt Core (models, tests, sources, exposures)
- Warehouse: DuckDB (local analytics + embedded OLAP)
- Streaming backbone: Apache Kafka (event ingestion, fanout)
- Data quality: Great Expectations (validation suites + checkpoints)
- BI / app layer: Streamlit (internal dashboards)

  scores gracefully. The raw data has: [schema]. Output: SQL for
  stg_oura__sleep.sql."
- "This Dagster asset is failing with: [error]. Root cause and fix? Provide the
  corrected asset definition and dependency spec."
- "Write a Great Expectations suite for a staging table that should always have:
  non-null user_id, sleep_score between 0-100, recorded_at within last 7 days."
- "Propose Kafka topic naming, partitions, and a retry strategy for ingestion
  of wearable device telemetry. Output: topic list + consumer pseudocode."

## Architecture Decisions with AI Rationale
- DuckDB over Postgres:
  - Claude recommended DuckDB for embedded analytics and zero-ops local runs.
  - Human confirmed that workloads are read-heavy and batch-centric.
  - DuckDB enables testing in CI without provisioning a server.
- Dagster over Airflow:
  - Claude highlighted Dagster’s asset lineage and type-checked inputs.
  - Human validated that asset-based modeling fits the medallion layout.
  - Dagster’s backfills and partitioned assets match data growth patterns.
- Medallion over flat:
  - Claude argued for clear quality boundaries (bronze/silver/gold).
  - Human agreed for auditability and easier backfill isolation.
  - Tiered models simplify governance and reduce blast radius.
- Great Expectations for quality:
  - Claude proposed GX for declarative, readable expectations and checkpoints.
  - Human confirmed GX integrates well with dbt and Dagster steps.
  - Expectations serve as both tests and documentation.

## Human Judgment Layer

  - Ryan chose domain-specific acceptable null rates per source.
  - Ryan set tolerance windows for late-arriving events.
  - Ryan decided which expectations are warnings vs hard failures.
- Business logic for correlations:
  - Ryan defined the causal vs correlational metrics presented in marts.
  - Ryan set aggregation windows for sleep vs activity comparisons.
  - Ryan defined the handling of outliers to avoid skewed results.
- Batch vs streaming trade-offs:
  - Ryan decided which sources remain batch for cost and stability.
  - Ryan chose where Kafka offsets are committed (at-least-once semantics).
  - Ryan set SLA boundaries for gold marts vs near-real-time views.

## Implementation Notes and Evidence of Use
- Claude drafts were always reviewed and modified by a human.
- SQL generated by Claude was validated with dbt tests and sample data.
- Asset dependencies suggested by Claude were confirmed by Dagster graph view.
- GX suites were exercised against real staging tables before promotion.
- Kafka consumer code was load-tested with synthetic data.
- Streamlit queries were benchmarked for DuckDB memory usage.

## Practical Limits Observed
- Claude occasionally overfit to generic dbt naming conventions.
- Claude missed a few Dagster edge cases (dynamic partitions).
- Claude suggested GX expectations that were too strict for raw data.
- Claude’s Kafka guidance needed adaptation to existing infra constraints.

## Auditability and Safety
- All AI-assisted changes were committed with descriptive messages.
- AI output was treated as a draft, not as source-of-truth.
- Human review emphasized schema evolution, data contracts, and SLAs.

## Summary
- Claude accelerated architectural drafting, SQL scaffolding, and test authoring.
- Humans retained ownership of correctness, domain logic, and operational trade-offs.
- The result is a medallion-structured pipeline with explicit quality gates,
  clear lineage, and repeatable validation steps.
